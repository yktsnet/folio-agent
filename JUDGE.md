# JUDGE — 技術選定と判断

本リポの技術選定の理由と、設計上の判断（なぜそれを選び、何を避けたか）を記録する。コードより先に「なぜ」を残すための文書。

**README への反映方針**: 本書は全判断の記録であり、README にはすべて載せない。README に展開するのは §1（CAG）・§2（OSS + dogfooding）・§3（Workers + LangGraph.js）・§5（知識の指定）・§9（配送と「LLM に渡した時点で公開」の原則）。それ以外（§4・§6〜§8・§10・§11）は README からは本書へのリンク参照に留める。

---

## 1. なぜ CAG（full-context）か — RAG でも Text-to-SQL でもない理由

- **判断**: ベクトル検索を持たず、**知識全量をビルド時に固めてシステムプロンプトに同梱する** full-context（CAG: Cache-Augmented Generation）方式を採る。
- **なぜ**: 知識源はポートフォリオサイト + 追加知識 + 記事数本で、LLM のコンテキストに余裕で収まる。この規模に検索基盤を足すのは過剰であり、`order-system-rag` で立てた「手法ありきのショーケースに堕さない」原則に反する。
- **物語の接続**: `order-system-migration`（質問の性質→SQL）、`order-system-rag`（質問の性質→RAG）に対し、本リポは**「知識の規模でツールを選ぶ」**第3の判断軸を実証する。3リポで「ツール選定の判断力」が一貫する。
- **境界の明示**: 知識が肥大してコンテキスト・コスト・応答品質が劣化したら RAG に切り替えるべきで、その境界が存在すること自体を README に記す。境界を知った上で手前側を選ぶのが本リポの主張。
- **避けたこと**: ベクトル DB・embedding パイプライン。壊れる部品が減り、OSS 利用者のセットアップも軽くなる。

## 2. なぜ独立 OSS + dogfooding か

- **判断**: portfolio-astro 内の機能としてではなく、**独立リポの汎用 npm パッケージ**として開発し、portfolio-astro を利用者第1号とする。
- **なぜ**:
  - ポートフォリオ開発とチャットボット開発を分離でき、Works が1つ増える。「OSS を作って自分で使っている」こと自体が展示価値。
  - サイト側は「Works を足すだけ」のフェーズに入っており、ボットの知識はビルド時取り込みで自動追従する。同期のための人手・AI の注意力に依存しない。
- **配布形態**: npm パッケージ（モノレポでウィジェットと Worker の2つ）。**テンプレートリポは不採用** — fork 後に本体の改善が届かず、OSS として育たない。
- **避けたこと**: v1 での過剰な汎用化。サポート対象を「dist を吐く静的サイト + Cloudflare Workers」に限定して README に明記する。一般化コスト（設定面の肥大・多フレームワーク対応・ドキュメント）は本体より重くなりがちで、利用者が現れてから払う。

## 3. なぜ Cloudflare Workers + LangGraph.js（TypeScript）か

- **判断**: バックエンドは Cloudflare Workers 上の **LangGraph.js** で組む。Python + 別ホストは採らない。
- **なぜ**:
  - **OSS としての採用されやすさ**: ポートフォリオサイトを持つ層は Astro/Next + Cloudflare/Vercel が主流。「npm install して wrangler deploy」で完結する TS 製が導入の壁を最小にする。Python 別ホストは利用者に常駐サーバを要求し、OSS として広がらない。
  - **プラットフォーム統合**: D1（ログ）・IP 取得（`CF-Connecting-IP`）・無料枠が Workers ネイティブで揃う。追加インフラゼロ。
  - **展示の幅**: Python LangGraph は `order-system-rag` で証明済み。同じ設計思想を **LangGraph.js でも書ける**ことを示す方が価値が増える。
- **なぜ LangGraph か（素の関数呼び出しにしない理由）**: 入力ガード → 意図ルーティング（考え方 / Works / 依頼）→ 生成 → ログ、と経路が分岐するグラフであり、`order-system-rag` と同じ道具で組むことで比較可能な展示になる。
- **トレードオフ**: LangGraph.js のバンドルサイズは Workers では注意が要る。使うのは StateGraph のコアのみに絞る。

## 4. なぜ Gemini 無料枠か

- **判断**: 生成は **Gemini（無料枠）** を既定とする。
- **なぜ**: 常時公開でコストゼロを維持する、既存2リポと同じ判断。生成エンジンが何かは主題（CAG という知識設計）を揺るがさない。
- **認識しているトレードオフ**: 無料枠は入力が学習に使われ得る。知識源は公開情報のみ、訪問者の入力もその前提で開示ページに明記するため許容。
- **無料枠の枯渇**: RPD 上限に達したら「本日の受付は終了しました + Contact 誘導」のフォールバックを返す。沈黙させない。

## 5. 知識ソースの指定 — なぜ「dist 走査 + URL グロブ」か

- **判断**: ビルド成果物 `dist/` の HTML を読み、利用者は **URL パスのグロブ**（include/exclude）で対象を選ぶ。
- **なぜ**: 「URL か ファイルパスか」の二択は合成できる。読み取りはファイルアクセス（クロール不要・ビルドと同居）、指定子は URL（利用者は自サイトの URL 構造だけ知っていればいい）。ソースのフレームワークに依存せず、dist を吐く静的サイトなら何でも動く。
- **追加知識**: サイトに構造化されていない情報は、**URL パスをミラーしたディレクトリ**（`knowledge/works/xxx.md`）に置く。どのページの補足かが構造で自明になり、明示的に「入れたものだけ入る」。
- **避けたこと**: 実行時クロール（デプロイ先への依存・遅延・robots 配慮が増える）、手書きの単一 JSON 知識ファイル（サイトとの二重管理になり、同期漏れが人手依存になる）。

## 6. ログと開示 — なぜ D1 か、なぜ同意ボタンを置かないか

- **判断**: 入力履歴を **D1（SQLite）** に記録する。開示は**チャット初回の一文 + 詳細ページ（別タブ）リンク**とし、同意ボタンは置かない。
- **なぜ D1**: チャットボットを持つ時点でログは品質改善（質問傾向・Contact 転換）に直結し、取らない理由がない。無料枠・スキーマ1つ・書き込み1行/会話で運用負荷ほぼゼロ。「DB は重い」は当たらない。
- **なぜ同意ボタンなしか**: IP + 入力内容は個人関連情報に当たり得るため通知はする。ただし一般のチャットウィジェットの水準に照らして、明示的同意まで要求する重さは UX を損なう。**黙って記録する多数派に対し、一文でも開示することが差別化**になる。
- **効果**: 「利用データの扱いを明記している」こと自体がショーケースとしてプロの仕事に見える。

## 7. レート制限 — なぜ D1 カウンタ・事後通知か

- **判断**: IP 単位で 10分3問・1日10回（環境変数で可変）。カウンタは**ログテーブルの集計を流用**する。事前宣言せず、超過時のみ通知する。
- **選択肢の比較**:
  | 選択肢 | 性質 | 採否 |
  |---|---|---|
  | **A. D1 ログ集計** | 既存テーブルの COUNT。仕組みが増えない | **採用** |
  | B. Workers Rate Limiting binding | 手軽だが固定ウィンドウ・日次上限が表現しづらい | 不採用 |
  | C. Durable Objects | 厳密な同時実行制御 | 不採用（この規模に過剰） |
- **なぜ事後通知か**: 事前に「1日10回まで」と掲げると窮屈な印象を与える。超過時に「上限に達しました。お急ぎは Contact へ」と返せば、**制限自体が Contact 誘導に転化**する。
- **認識している限界**: 企業 NAT・モバイル回線では複数人が同一 IP になる。展示会等で見せる場面を考え、数字は環境変数で即調整できるようにする。IP 偽装・VPN での回避は防がない（ショーケースの被害想定では Gemini 無料枠の消費が上限）。

## 8. UI — なぜ自前の軽量ウィジェットか

- **判断**: 既製チャットウィジェット（Intercom 系・埋め込み SaaS）を使わず、自前のフレームワーク非依存コンポーネントを書く。
- **なぜ**: 既製品は重く、無音・非ポップアップの要件（クリックするまで一切主張しない）を細かく制御しづらい。ポートフォリオでは**ウィジェット自体が作品**であり、軽さがそのまま展示価値になる。
- **要件として固定**: 自動ポップアップなし・初回吹き出しなし・未クリック時の通信なし（知識もモデルも呼ばない）。
- **テーマの受け渡し（Issue 01 で追加）**: 配色・フォントは **CSS カスタムプロパティ6トークン**（`--folio-agent-surface / text / muted / accent / accent-contrast / font`）で受け、フォールバックに既定デザインを持つ。JS の設定 API やビルド時テーマではなくカスタムプロパティを採ったのは、Shadow DOM でも host から継承される標準機構であり、利用側がグローバル CSS 数行で済み、未設定なら見た目が変わらないため。「どのサイトにも馴染む仕組みを作り、自サイト（poi パレット）はその実証第1号」という位置づけ。
- **回答表示はプレーンテキスト固定**: Markdown レンダラを積まず、記法の禁止はプロンプト側（共通前文）で行う。レンダラは重さと XSS 面の管理コストを増やし、受付チャットの短文回答に対して割に合わない。

## 9. 知識の配送 — なぜサイトと同一デプロイに同梱するか（v1）

- **判断**: 生成した知識ファイルは、v1 では**サイトと同じデプロイに同梱**し、チャット API もサイトと同一プロジェクトの Function として動かす（案D）。
- **選択肢の比較**:
  | 選択肢 | 仕組み | 採否 |
  |---|---|---|
  | **D. 同一デプロイ同梱** | サイトのビルドで知識を生成しそのまま同梱。API は Pages Functions | **採用（v1）** |
  | C. 静的アセット公開 + fetch | 知識を dist に含めて公開 URL 化し、独立 Worker が fetch | 将来追加（Cloudflare 外のサイト向け） |
  | A. サイト CI から KV へ PUT | CI に CF API トークンを置き KV 経由で受け渡し | 不採用（C の疎結合をより重いセットアップで買うだけ） |
  | B. Worker 再デプロイ同梱 | サイト更新のたびに別リポの Worker をデプロイ | 不採用（継ぎ目が CI のクロスリポ連携に移動するだけ） |
- **なぜ D**: 継ぎ目（知識の生成場所と消費場所のズレ）が設計ごと消える。デプロイ1つ・知識とサイトが原子的に一致・追加トークンや KV が不要で、v1 スコープ（静的サイト + Cloudflare）と完全に整合する。リポ・パッケージ・グラフ設計の層で分離は成立しており、実行の同居は独立 OSS の看板を損なわない。
- **C を残す理由**: サイトのホスト先が Cloudflare でない利用者には D が成立しない。生成段階（CLI → 知識ファイル1つ）は D/C で完全に共通のため、配送段階だけ差し替え可能な境界で切っておけば、C は後からインターフェース変更なしで追加できる。v1 では実装せず、README に構造と境界のみ記す。
- **管理の原則（C/D 共通）**: **LLM のコンテキストに渡した時点で公開扱い**とする。プロンプト抽出は完全には防げず、知識ファイルが URL で落とせるか（C）否か（D）は秘匿性の差にならない。したがって knowledge/ に書いてよいのは「原文で引用されても構わないもの」だけ。この境界は配送方式の選択に依存しない。
- **効果**: Bot の知識の正は「include したページ + knowledge/（+ Zenn）」に決定的に定まり、生成物を見れば全量確認できる。RAG と違い「何を知っているか」が間接化しないのは CAG の利点そのもの。

## 10. 経路分類 — なぜ LLM 判定ではなくキーワードによる決定的分岐か

- **判断**: 考え方 / Works / 依頼の3経路振り分け（`route.ts`）は、LLM 呼び出しではなく**キーワードマッチによる決定的分岐**で行う。
- **なぜ**: 経路判定に失敗しても実害は小さい（generate 側の知識プロンプトは全経路共通で、違うのは口調・振る舞いの指示だけ）。この程度の分岐に追加の LLM 呼び出し（レイテンシ・コスト・失敗点）を足す理由がない。決定的であることはテストのしやすさにも直結する（`route.test.ts` はキーワード→経路の対応表そのもの）。
- **認識している限界**: キーワードに現れない依頼相談（婉曲表現等）は thoughts/works に流れうる。実害は小さいと判断した上での割り切りであり、誤判定の兆候が増えたら LLM 判定への切り替えを検討する。
- **境界**: この判断は経路分類という「軽い意思決定」に限定される。回答生成そのもの（知識に基づく応答文の組み立て）は最初から Gemini（LLM）が担っており、CAG の主題（知識の与え方）とは別軸の判断である。

## 11. モノレポ管理 — なぜ npm workspaces か（pnpm 不採用）

- **判断**: モノレポは **npm workspaces** で管理する。当初 pnpm workspaces を予定していたが変更した。
- **なぜ**: widget / handler の2パッケージ・依存も軽い規模では、pnpm の利点（ディスク効率・厳密な依存解決）がほぼ効かない。npm なら Node があれば動き、コントリビュータにも制約の強い開発環境（Nix 等）にも追加ツールが要らない。publish される成果物はどちらで開発しても同一。
- **避けたこと**: `npx pnpm` 経由での運用（README・CI・CLAUDE.md の全コマンドが非標準になる）と、corepack のグローバル書き込みを前提にした手順（Nix 管理環境で権限エラーになる）。
- **切り替え基準**: パッケージ数が増えて依存の hoisting 事故やインストール時間が実害になったら再検討する。この規模で先回りはしない。
