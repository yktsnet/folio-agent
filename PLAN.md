# folio-agent — 構想メモ (PLAN)

ポートフォリオサイトに組み込む**受付エージェント（チャットボット）を独立 OSS として開発する**プロジェクト。サイトのビルド成果物を知識源とし、full-context（CAG）方式で回答する。技術選定の理由は [JUDGE.md](JUDGE.md) を参照。

## 1. 目的

- サイト訪問者の質問（作者の考え方・Works の解説）に、**サイトに書いてあることだけから**答える。
- 仕事の依頼・相談の気配を検知したら Contact へ誘導する。
- `order-system-migration`（Text-to-SQL）・`order-system-rag`（RAG）に続く第3の実証 —— **「知識の規模でツールを選ぶ」**（小さな知識に検索は不要、全部渡す）。
- 独立 OSS として開発し、`portfolio-astro` が利用者第1号として組み込む（dogfooding）。必須機能ではなくショーケース。

## 2. スコープの位置づけ（3リポの物語）

| 観点 | 課題 | 担当リポ |
|---|---|---|
| 質問の性質（構造化） | 集計・ランキング | `order-system-migration`（Text-to-SQL） |
| 質問の性質（非構造） | 根拠付き検索 | `order-system-rag`（RAG） |
| **知識の規模（小）** | 検索せず全量をコンテキストへ | `folio-agent`（本リポ・CAG） |

- 知識が一定量を超えたら RAG に切り替えるべき境界が存在すること自体が CAG の物語の一部。判断基準を README に明記する。
- ポートフォリオ開発とチャットボット開発を**リポとして分離**する。本リポの成果物は汎用 npm パッケージであり、portfolio-astro 固有の知識を持たない。

## 3. 体験設計

### ウィジェット（フロント）

- サイト右下に常駐。**クリックするまで無音**。自動ポップアップ・初回吹き出しなし。
- 既製ウィジェットは使わず、自前の軽量コンポーネント（フレームワーク非依存）。
- チャット初回表示に「入力内容は品質改善のため記録されます」の一文 + 利用データの扱いページ（別タブ）へのリンク。同意ボタンは置かない。

### 回答の3経路（LangGraph ルーティング）

| 経路 | 内容 |
|---|---|
| 考え方 Q&A | サイト・追加知識に書かれた作者の考え方から回答。**書いてないことは推測せず** Contact へ |
| Works 解説 | Works の内容を解説。対応する Zenn 記事があればリンクを添える |
| 依頼・相談 | Contact ページへ誘導（オプションでオン/オフ） |

### レート制限

- IP（`CF-Connecting-IP`）単位で **10分3問・1日10回**。数字は環境変数で調整可能。
- 事前宣言はせず、超過時に「本日の上限に達しました。お急ぎの場合は Contact へ」と返して誘導に転化する。

## 4. 知識ソースの設計

サイトが Git 管理 + CI/CD デプロイされている前提と相性を取り、**同期作業そのものを無くす**。

1. **サイト本体**: ビルド時に `dist/` の HTML 全体を走査。読ませるページは利用者が **URL パスのグロブ**で選ぶ:
   ```json
   { "include": ["/", "/works/**", "/about"], "exclude": ["/works/draft-*"] }
   ```
2. **追加知識**: サイトに構造化されていない情報（考え方・仕事の受け方等）は、**URL パスをミラーしたディレクトリ**に Markdown で明示配置する（例: `knowledge/works/order-system-rag.md`）。
3. **Zenn 等の外部記事**: オプション。取り込めば Works に紐づく解説の厚みが増す。
4. ビルドのたびに 1〜3 を結合してシステムプロンプト用の知識ファイルを生成する。Works 追加・記事追加は次のデプロイで自動反映。

## 5. アーキテクチャ

```
利用者のサイト（静的ビルド）
  └── ビルド時: folio-agent CLI が dist/ + knowledge/ + (Zenn) → 知識ファイル生成
ブラウザ
  └── ウィジェット → POST /api/chat → サイトと同一デプロイの Function（Pages Functions）
        LangGraph.js StateGraph:
          input_guard → route(考え方 / Works / 依頼) → generate → log
        ├── 知識: ビルド時に同梱（配送案D。詳細は JUDGE §9）
        ├── LLM: Gemini（無料枠）
        └── D1: 入力履歴ログ + レート制限カウンタ
```

- モノレポで npm パッケージ2つ: **ウィジェット**（フロント）と **チャットハンドラ + 取り込み CLI**。
- 知識の配送は v1 では**サイトと同一デプロイに同梱**（案D）。生成段階と配送段階を分離した設計にし、Cloudflare 外のサイト向けの配送（案C: 知識を静的アセット公開 + 独立 Worker が fetch）は将来追加できる構造と境界を README に明記する。
- ログテーブルをそのままレート制限のカウンタに流用する（別の仕組みを持たない）。

## 6. 設計上の境界（Out-of-Scope）

- **v1 のサポート対象は「dist を吐く静的サイト + Cloudflare Workers」のみ**。多フレームワーク・多ホスト対応はしない（README に明記）。汎用化は利用者が現れてから。
- 書いてないことに答えない（無回答 + Contact 誘導）。
- 認証・認可、会話の永続セッション、多言語対応は対象外。
- Gemini 無料枠は入力が学習に使われ得る。知識源は公開情報のみなので許容（利用データの扱いページに記載）。
